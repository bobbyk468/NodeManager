Q4. Failure Modes — THREE SPECIFIC RISKS
══════════════════════════════════════════════════════════

1. FORMAT ADHERENCE DEGRADATION
   After a 2,000-token think trace, open LRMs frequently fail to output
   strict JSON. Backend will crash on parsing errors.
   FIX: Add regex fallback + constrained decoding (Outlines or Guidance)
        to enforce {"valid": bool, "reasoning": str} output format.

2. OVERTHINKING PENALTY
   LRM may reject valid standard answers by reasoning into obscure
   technical edge cases.
   FIX: Add confidence threshold — if verifier rejects but concept
        coverage is high (>80%), override with a human-review flag
        rather than automatic penalty.

3. LATENCY SPIKES
   Inference: ~1s (Flash) → 15–40s (LRM) per answer.
   FIX: Run Stage 3 asynchronously in batch mode, not real-time.
        Pre-compute and cache in eval_results.json (same pattern as
        existing batch pipeline).

ACTION: Implement all three safeguards before running ablation.

Q7. CRITICAL FLAW — RAW TRACE IS NOT EDUCATOR-READABLE
══════════════════════════════════════════════════════════

This is the single most important finding.

Unfiltered <think> traces are rambling, repetitive, and contradictory
as the model tests hypotheses. Showing raw text to an instructor may
DECREASE trust and INCREASE cognitive load.

REQUIRED: A computational trace-parsing step between the LRM output
and the dashboard must:
  1. Extract atomic reasoning steps (split trace into structured assertions)
  2. Classify each step: [SUPPORTS / CONTRADICTS / UNCERTAIN] re: KG edge
  3. Link each classified step to the specific KG node/edge it references
  4. Discard hypothesis-testing branches (show only final conclusion path)

This trace parser is itself a research contribution in the paper.

ACTION: Design TraceParsing module as Stage 3b in the pipeline.
        This becomes Section III.D in the architecture section.
