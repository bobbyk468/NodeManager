# Gemini Review: AGENT_EVALUATION_GUIDE.md (v13 Feedback)

## Q1: Code Correctness & Best Practices
**Summary:** The code contains several critical missing imports and stale closure risks that will cause runtime errors or dropped telemetry.
**Findings:**
- `stability_analysis.py`: Missing `import os`. The script will crash on `os.path.exists(ablation_path)`.
- `AnswerViewPanel.tsx`: `tracePanelOpen` and `kgPanelOpen` are used in the `useEffect` cleanup function but are not stored in `useRef`. They will be stale closures, always reflecting their value from when the component first mounted.
- `AnswerViewPanel.tsx`: Calls `useStudyContext()` but Section 7 states the context is `useDashboardContext()`.
- `seeding.service.ts`: `readFileSync` in the constructor is synchronous. While acceptable for startup, NestJS recommends `OnModuleInit` for initial I/O.
**Recommendations:**
1. Add `import os` to `stability_analysis.py`.
2. Add `tracePanelOpenRef` and `kgPanelOpenRef` to the `AnswerViewState` ref in `AnswerViewPanel.tsx`.
3. Change `useStudyContext()` to `useDashboardContext()` in `AnswerViewPanel.tsx`.
4. Move `loadSeedRegistry()` to `onModuleInit()` in `SeedingService`.
**Severity:** Critical

**Severity:** High

## Q3: Data Schema & Event Log Validation
**Summary:** There are major schema mismatches between the frontend payload generation and the backend NestJS DTOs, which will cause `class-validator` to reject valid events.
**Findings:**
- `RubricEditPayloadDto` requires `@IsInt() trace_gap_count` and `@IsNumber() grounding_density`, but these are not included in the frontend `onRubricEdit` payload snippet in Section 3 (Rule 2).
- The JSON example in Section 6 shows `dwell_time_ms` and `benchmark_case` inside a `rubric_edit` event payload. However, the DTO only defines these for `AnswerDwellPayloadDto`. `whitelist: true` will strip them, or validation will fail.
- `RubricEditPayloadDto` is missing `source_contradicts_nodes_60s` which is generated by the frontend in Rule 2.
**Recommendations:**
1. Update `RubricEditPayloadDto` to include `@IsOptional() @IsArray() @IsString({ each: true }) source_contradicts_nodes_60s?: string[]`.
2. Add `trace_gap_count` and `grounding_density` to the frontend `onRubricEdit` payload example.
3. Fix the JSON example in Section 6 so `dwell_time_ms` and `benchmark_case` are not mixed into a `rubric_edit` payload.
**Severity:** Critical

## Q7: Suitability for Agent Execution
**Summary:** An agent will likely get stuck during Phase 4 (Validation) because the mock events generated by the frontend will be rejected by the backend DTOs due to missing fields.
**Findings:**
- The strictness of `class-validator` combined with missing fields (`trace_gap_count` in frontend, `source_contradicts_nodes_60s` in backend) guarantees a 400 Bad Request during the integration test. The agent will spin trying to fix the validation pipe.
**Recommendations:**
1. Provide a perfectly matched, complete example of `RubricEditPayload` in both the frontend code snippet and the backend DTO snippet.
2. Add an explicit "Assumptions" note that the agent should ensure frontend payload keys perfectly match DTO properties before running tests.
**Severity:** Critical
